# 재원기간 기산일을 "첫 출석일"로 변경

- 작성일: 2026-05-29
- 범위: 크로스앱 (`@impact7/shared` + impact7newDSC + impact7DB)
- 조율: RULES상 크로스앱 변경이므로 impact7DB에서 조율·배포

## 배경 / 문제

재원기간(`deriveTenure`)은 `history_logs`의 `'신규'`/`'재등원'` 이벤트 날짜를
현재 재원 구간의 시작으로 잡는다. 이 날짜는 "등원예정을 등록한 행위 시점"이라,
**한 번도 등원하지 않은 학생(예: 임채윤, 선화예/고2 — 수업 시작일 2026-07-14 미래,
status 등원예정)도 재원 N일로 카운트**되는 문제가 있다.

레벨기간(`deriveLevelPeriod`)은 이미 enrollment의 `start_date` 기준이라 미래면
`'등원예정'`으로 맞게 나오지만, 재원기간은 등록일부터 카운트되어 두 값이 어긋난다.

## 목표

재원기간의 기산일을 **현재 재원 구간의 첫 실제 출석일**로 변경한다.

- "출석"의 정의 = `isAttendedStatus`: `'출석' | '지각' | '조퇴'` (결석·미확인·등원전 제외)
- 출석 기록이 0건이면 헤더에 **`'등원예정'`** 표시
- 퇴원 후 재등원생은 **마지막 재등원/신규 이벤트 이후**의 첫 출석일 (= "지금 다니는 구간")
- `@impact7/shared`의 SSoT 로직을 바꿔 impact7DB·impact7newDSC 양 앱이 동일 동작

## 비목표 (YAGNI)

- 레벨기간(`deriveLevelPeriod`) 로직 변경 — 이미 의도대로 동작
- 최초(맨 처음) 첫 출석 누적 방식 — 현재 구간 기준만 사용
- 첫 출석일을 학생 문서에 캐싱하는 비정규화 — 학생 1명 조회라 온디맨드로 충분
- 미등원 표시에 `D-n` 카운트다운

## 설계

### 1. `deriveTenure` 시그니처·반환 변경

위치: `@impact7/shared` 소스 레포 `~/projects/impact7-shared/history-classifier.js`

```js
// before
deriveTenure(logs, getDate) → { start, end }

// after
deriveTenure(logs, getDate, attendedDates) → { start, end, startEvent }
```

- `attendedDates`: 해당 학생의 비결석 출석 날짜 배열. 호출부가 `daily_records`의
  `date`('YYYY-MM-DD' 문자열)를 그대로 넣어 주입한다.
- 타입 통일: 내부에서 `startEvent`(Date)와 비교할 때 `startEvent`를 KST 기준
  'YYYY-MM-DD' 문자열로 변환한 뒤 **문자열 사전순 비교**로 ≥ 판정한다
  (`deriveLevelPeriod`가 이미 쓰는 방식과 일관). `attendedDates`는 정렬 후 첫 만족값.
- 내부 동작:
  1. 기존 로직대로 logs를 `classifyHistory`로 분류해 현재 재원 구간의
     시작 이벤트일 `startEvent`(마지막 `'신규'`/`'재등원'`)와 `end`(`'퇴원'`)를 계산
  2. `start` = `attendedDates` 중 `startEvent` **이후(≥)** 가장 이른 날. 없으면 `null`
  3. `{ start, end, startEvent }` 반환
- 의미 구분 (호출부 표시용):
  | 상태 | 조건 | 표시 |
  |------|------|------|
  | 이력 없음 | `startEvent == null` | `'—'` |
  | 미등원 | `startEvent != null && start == null` | `'등원예정'` |
  | 재원 중 | `start != null` | `시작 ~ 현재/종료` |

`attendedDates`는 필수 인자로 둔다(양 앱 동시 수정). 옛 2-인자 호출이 남아 있으면
`start`가 `null`로 떨어지므로, 마이그레이션 시 모든 호출부를 함께 고친다.

### 2. 표시 로직 (양 앱 각각)

- impact7newDSC: `student-detail.js`의 `formatTenure(start, end, student)` 를
  `{ start, end, startEvent }` 기준으로 수정.
  - `startEvent && !start` → `'등원예정'`
  - `!startEvent` → `'—'`
  - `start` → 기존 `_fmtTenureDate(start) ~ (end/현재/종강)` 포맷 유지
- impact7DB: `app.js:4329` 부근 deriveTenure 결과 표시부에 동일 분기 적용.

### 3. 호출부 데이터 로딩 (양 앱 각각)

- impact7newDSC: `student-detail.js`의 `fillTenure`(라인 ~100-120).
  기존 history_logs 쿼리에 더해:
  ```js
  const aq = query(collection(db, 'daily_records'), where('student_id', '==', studentId));
  const asnap = await getDocs(aq);
  const attendedDates = [];
  asnap.forEach(d => {
      const r = d.data();
      if (isAttendedStatus(r?.attendance?.status) && r.date) attendedDates.push(r.date);
  });
  ```
  history_logs 쿼리와 병렬(`Promise.all`)로 실행해 라운드트립을 늘리지 않는다.
- impact7DB: `app.js:4329` 호출부에 동일 패턴.
- `isAttendedStatus`는 현재 impact7newDSC의 `attendance.js`에만 있다.
  양 앱이 같은 정의를 쓰도록 **`@impact7/shared`로 승격**하거나, 각 앱에서 동일 상수를
  쓴다 (구현 단계에서 결정 — shared 승격을 우선 검토).

### 4. 엣지·성능

- 재등원생: `startEvent` 이후 필터로 "현재 구간 첫 출석"이 자동 처리.
- `daily_records`는 `student_id` 단일 `where`만 쓰므로 기본 인덱스로 충분 (date range 미사용).
- `startEvent`보다 이른 출석 기록만 있는 비정상 케이스 → `start = null` → `'등원예정'` (안전).
- 학생 1명 헤더 조회. 출석 기록 수백 건 수준이라 history_logs 200건 쿼리와 동급 비용.

### 5. 테스트

`~/projects/impact7-shared`에 `deriveTenure` 단위 테스트 추가:
1. 신규 + 이후 출석 있음 → start = 첫 출석일
2. 신규 + 출석 0건 → start = null, startEvent != null (등원예정)
3. 재등원 → 마지막 재등원 이후 첫 출석일 (그 이전 출석은 무시)
4. 이력 없음(logs 빈 배열) → startEvent = null, start = null
5. 퇴원 → end 세팅, start는 해당 구간 첫 출석일

### 6. 크로스앱 작업 순서

1. `~/projects/impact7-shared`: `deriveTenure` 수정 + 테스트 + (필요시) `isAttendedStatus` 승격 → 버전 태그 (예: v1.11.0)
2. impact7DB: shared 버전 bump + `app.js` 호출부/표시부 수정 + 빌드
3. impact7newDSC: shared 버전 bump + `student-detail.js` 호출부/표시부 수정 + 빌드
4. 양 앱 배포 (impact7DB 조율)

## 검증 시나리오

- 임채윤(선화예/고2): 출석 0건 → 헤더 재원기간 `'등원예정'` 표시 확인
- 정상 재원생: 첫 출석일 ~ 현재로 표시, 등록일이 아닌 첫 출석일 기준인지 확인
- 재등원생: 현재 구간 첫 출석일부터 표시되는지 확인
